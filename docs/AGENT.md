# AGENT

> 이 문서는 이 프로젝트의 단일 운영 문서다.
> 비사소한 작업을 시작하기 전에 항상 가장 먼저 이 문서를 다시 읽는다.

## 1. 이 문서의 역할

- 이 문서는 프로젝트 운영 방법, 작업 모드, 장시간 작업 규칙, 문서 규칙, 산출물 규칙, 네이밍 규칙을 통합 관리한다.
- 이 문서는 정책과 방법만 다룬다.
- 프로젝트 고유의 기술 스택, 모델 선택, 하드웨어 목표, 아키텍처 상세는 이 문서에 오래 쌓지 않는다.
- 프로젝트의 현재 상태, 현재 전역 결정, 최근 작업 흐름은 `docs/logbook.md`가 맡는다.
- 모듈별 상세 아키텍처와 고정 결정은 각 모듈 `README.md`가 맡는다.

## 2. 항상 읽는 순서

- 비사소한 작업에서는 아래 순서를 항상 다시 통과한다.
  1. `docs/AGENT.md`
  2. `README.md`
  3. `docs/logbook.md`
  4. 최신 `docs/logbook_archive/logbook_*.md` 1개
  5. 관련 모듈 `README.md`
  6. 필요한 경우에만 관련 모듈의 날짜 prefix 문서
- 이 순서는 아래 상황마다 다시 적용한다.
  - 새 작업 시작
  - 단계 전환
  - 결과 정리
  - 문서 구조 변경
  - 산출물 정리 또는 삭제 판단
  - 커밋 전
  - 푸시 전
  - 후속 실행 시작 전
- 코드 수정이나 문서 수정이 없는 `질문 응답만 하는 턴`도 예외가 아니다.
- 어떤 질문이든 답변 초안을 만들기 전에 최소한 `docs/AGENT.md`는 다시 읽고 현재 모드, prefix, 말투 규칙을 맞춘다.
- 시간이 부족하거나 질문이 짧다는 이유로 `AGENT` 재확인을 생략하지 않는다.
- 새 기능 추가, 폴더 이동, 레이어 경계 판단이 포함된 작업은 `README.md`의 `새 기능을 어디에 둘까` 섹션을 반드시 다시 확인한다.
- 비공개 원격, 비공개 대용량 자산, 자격증명, 외부 미러, private publish 운영이 포함된 작업은 sibling `../secrets/README.local.md`와 관련 하위 로컬 문서를 함께 다시 읽는다.

## 3. 현재 문서 구조

- `docs/AGENT.md`
  - 프로젝트 운영 정책과 작업 방법의 단일 기준
- `docs/logbook.md`
  - 프로젝트 레벨 현재 스냅샷, 전역 결정, 활성 체크리스트, 최근 로그의 단일 기준
- `docs/logbook_archive/`
  - 이전 active logbook 보관소
- `README.md`
  - 프로젝트 소개, 전체 구조, 프로젝트 전역 고정 메모, root shared asset 진입점
- `<module>/README.md`
  - 해당 모듈의 상세 소개, 고정 결정, 경로, 자산 ownership, 실행 절차
- `<module>/docs/logbook.md`
  - 해당 모듈의 현재 상태, 최근 변경, 다음 작업의 단일 기준
- `<module>/docs/logbook_archive/`
  - 해당 모듈 logbook archive
- `<module>/docs/`
  - 날짜 prefix로 누적되는 환경/보고서 문서
- `<module>/results/`
  - 실제 대용량 로컬 실행 산출물
- `assets/`
  - root `README.md`가 직접 참조하는 프로젝트 전역 공용 자산
- `../secrets/README.local.md`
  - 비공개 원격, 비공개 자산, 자격증명, 외부 미러 운영의 로컬 전용 시작 문서

## 4. 문서별 단일 역할

- `docs/AGENT.md`
  - 운영 정책
- `docs/logbook.md`
  - 프로젝트 레벨 현재 상태와 로그
- `README.md`
  - 프로젝트 소개, 전체 구조, 메인 차원의 고정 메모
- `../secrets/README.local.md`
  - tracked repo 밖에서만 관리하는 비공개 운영 세부와 자격증명 메모
- `<module>/README.md`
  - 모듈별 상세 기준
- `<module>/docs/logbook.md`
  - 모듈별 현재 상태, 활성 체크리스트, 최근 기록
- `<module>/docs/보고서/`
  - 사용자가 별도로 요청했거나 외부 공유가 필요한 경우에만 만드는 요약 문서
- `<module>/docs/환경/`
  - 설치, 재현, 운영 절차
- 같은 내용을 여러 상위 문서에 반복하지 않는다.
- 프로젝트 레벨 현재 상태를 바꾸면 `docs/logbook.md`를 먼저 갱신한다.
- 모듈별 상세 기준이 바뀌면 해당 모듈 `README.md`를 먼저 갱신한다.
- 특정 모듈에 속하지 않는 프로젝트 전역 구조 메모나 root 공용 자산 설명은 `README.md` 아래에 누적한다.
- 비공개 원격, private artifact storage, provider별 토큰, repo id, publish cadence 같은 운영 세부는 tracked 문서에 쓰지 않고 `../secrets/README.local.md`와 그 하위 로컬 문서에만 둔다.
- private artifact allowlist, provider별 publish/sync 스크립트, 실제 업로드 대상 목록 같은 자동화 세부도 tracked 문서에 쓰지 않고 `../secrets/README.local.md`와 그 하위 로컬 문서에만 둔다.

## 5. 작업 모드

### 5-1. 공통 원칙

- `협업 모드`, `에이전트 모드` 모두 사용자에게는 기본적으로 친절한 말투로 안내한다.
- 단, 과장된 추임새나 불필요하게 장황한 표현은 피하고, 필요한 정보는 분명하게 전달한다.
- 기본 설명 언어는 한국어로 두고, 가능하면 비개발자도 바로 이해할 수 있는 쉬운 말부터 쓴다.
- tracked 문서는 `AI가 대신 남긴 메모`처럼 쓰지 않고, 프로젝트 작성자나 운영자가 직접 정리한 기술 문서처럼 쓴다.
- `AI가 정리했다`, `Codex가 남긴다`, `에이전트가 분석했다`처럼 작성 주체가 도구로 드러나는 표현은 tracked 문서에서 쓰지 않는다.
- README, 모듈 README, logbook는 가능하면 `프로젝트가 무엇을 제공하는지`, `현재 무엇이 확인되었는지`, `어떻게 실행하는지`가 바로 읽히게 적고, 작업 도우미의 흔적처럼 보이는 문장은 피한다.
- 개발 용어, 영어 식별자, 축약어는 꼭 필요할 때만 쓰고, 먼저 쉬운 한국어 설명을 붙인 뒤 사용한다.
- 문서와 답변에서는 `fallback`, `verdict`, `mapping`, `surface`, `contract`처럼 익숙하지 않을 수 있는 용어를 압축해서 반복하지 말고, 가능한 한 `대체 응답`, `판단 결과`, `연결`, `입력 범위`, `입출력 약속`처럼 풀어쓴다.
- 사용자가 이해하기 어렵다고 지적한 표현은 같은 턴 안에 바로 더 쉬운 표현으로 바꾸고, 반복해서 쓰지 않는다.
- 현재 실제로 무엇이 돌고 있는지 말할 때는 대화 맥락이 아니라 `ps`, `pid`, 상태 파일, 진행률 파일을 먼저 확인한다.
- 사용자의 질문에 답변하기 전에는 현재 턴의 첫 확인 문서로 항상 `docs/AGENT.md`를 다시 읽는다.
- 질문이 단순 설명 요청이더라도, 답변 prefix, 말투, 쉬운 한국어 원칙, 현재 모드 규칙은 `AGENT`를 다시 읽고 맞춘다.
- final 답변 전에는 반드시 `현재 작업 모드`, `답변 prefix`, `active checklist 갱신 여부`, `canonical 문서 반영 여부`를 한 줄 체크포인트로 다시 확인한다.

### 5-2. 협업 모드

- 기본 모드는 `협업 모드`다.
- 사용자의 중간 질문, 선택, 피드백을 자주 반영하면서 진행한다.
- 이 모드에서도 비사소한 작업 단위의 첫 게이트는 항상 이 문서다.
- 장시간 작업은 함부로 끊지 않지만, 다음 큰 단계로 넘어갈 때는 사용자의 현재 의도를 더 자주 반영한다.
- 이 모드의 답변 prefix는 `협업)`다.

### 5-3. 에이전트 모드

- 사용자가 명시적으로 `에이전트 모드`를 선언하면, 그 시점부터는 문서와 실제 실행 상태를 기준으로 스스로 이어서 진행한다.
- 이 모드에서는 사용자가 자리를 비웠을 가능성을 기본 전제로 둔다.
- 명시적인 중단 지시가 없는 한 이미 시작한 변환, 검증, benchmark, 배포 준비 작업을 끊지 않는다.
- 중간 질문은 기본적으로 상담이나 상태 확인으로 해석하고, 답변 직후 원래 작업 흐름으로 즉시 복귀한다.
- 장시간 작업이 끝나면 다음 단계로 자연스럽게 이어가되, 그 전에 항상 `README.md`와 `docs/logbook.md`를 다시 읽고 기준을 맞춘다.
- 이 모드의 답변 prefix는 `AI)`다.

## 6. 프로젝트 고유 기술 내용은 다른 문서가 맡는다

- 이 문서는 다른 프로젝트에도 바로 가져다 쓸 수 있도록 프로젝트 고유 기술 내용을 최소화한다.
- 프로젝트의 제품 목표, 하드웨어 타깃, 전체 파이프라인, 공용 아키텍처 그림, 전역 구조 메모는 `README.md`에 둔다.
- 프로젝트의 현재 상태, 현재 우선순위, 최근 작업 흐름은 `docs/logbook.md`에 둔다.
- 모듈별 모델 선택, 비채택 후보, 상세 경로, 자산 ownership, 재현 절차는 각 모듈 `README.md`에 둔다.
- 즉 `AGENT`는 운영 방법만, `README`는 프로젝트 전체 설명만, `logbook`은 현재 truth와 로그만 담당한다.

## 7. 핵심 개발 원칙

### 7-1. 역할 분리를 먼저 지킨다

- 운영 규칙, 현재 상태, 프로젝트 소개, 모듈 상세 기준이 서로 문서를 침범하지 않게 유지한다.
- 프로젝트 전역 메모가 특정 모듈에 들어가면 안 되고, 모듈 상세 결정이 `AGENT`나 `logbook`에 오래 눌러앉아도 안 된다.

### 7-2. 단순한 모듈 인터페이스

- 각 요소기술은 독립 폴더와 공통 진입점으로 관리한다.
- backend가 여러 개여도 상위 사용법은 하나로 유지한다.
- 데모 파일과 실제 모듈 파일은 분리한다.

### 7-3. 오버엔지니어링 금지

- 처음부터 과도한 추상화, 미사용 옵션, 추측성 fallback을 넣지 않는다.
- 먼저 주경로를 단순하게 완성하고, 실제 문제를 본 뒤 필요한 만큼만 확장한다.

### 7-4. 계획 기반 실행

- 작업은 시작 전에 계획을 세우고 순서대로 진행한다.
- 중간에 계획이 바뀌면 이유를 남긴다.
- 장시간 변환, 양자화 검증, 실기기 benchmark, 다단계 실험처럼 나중에 컨텍스트가 많이 압축될 수 있는 작업은 실행 전에 계획을 문서에 남긴다.
- 계획 문서는 최소 `목표`, `단계`, `성공 기준`, `중단 조건`, `산출물 경로`를 포함한다.
- 비사소한 개발 작업은 항상 `체크리스트 기반`으로 진행한다.
- 단, 체크리스트를 매번 새 `보고서` 파일로 만들지는 않는다. 기본 위치는 `docs/logbook.md`다.
- 기본 원칙은 아래다.
  - 작은 실행과 짧은 후속 작업은 해당 모듈 `docs/logbook.md` 안의 `현재 활성 체크리스트`로 관리한다.
  - 여러 파일을 건드리지만 범위가 아직 작고, 당일 안에 닫을 가능성이 높은 작업도 우선 `docs/logbook.md`를 기본 체크리스트 자리로 쓴다.
  - 큰 단계 전환, 장시간 작업, 마일스톤이라도 보고서는 자동으로 만들지 않는다.
  - 별도 `docs/보고서/YYMMDD_HHMM_이름.md`는 사용자가 명시적으로 요청했거나 외부 공유용 요약이 필요할 때만 만든다.
- 체크리스트 기반 실행 문서는 큰 phase 설명만 적고 끝내지 않고, `이번 작은 실행 단위`에서 실제로 손댈 파일, 확인할 입력, 만들어질 산출물, 검증 명령, 다음 단계 연결점을 checklist 항목으로 쪼개 적는다.
- 체크리스트 문서에는 최소 아래를 함께 적는다.
  - 이번 실행의 목표와 비범위
  - 수정 대상 파일과 각 파일의 역할
  - 생성되거나 갱신되는 산출물 경로
  - 각 산출물이 다음 어떤 모듈, 다음 어떤 단계, 어떤 런타임 입력으로 연결되는지
  - 검증 방법과 완료 조건
- 체크리스트가 `docs/logbook.md` 안에 있을 때도, 상위 계획 문서와 관련 모듈 `README.md`, 필요하면 `docs/logbook.md`를 같이 링크하거나 바로 옆에 적어 문서 이동이 끊기지 않게 한다.
- 별도 체크리스트 문서와 상위 계획 문서는 문서 안에서 서로 링크로 연결한다.
- 별도 작은 실행 문서는 최소 아래 상위 문서를 직접 링크해야 한다.
  - 자신이 속한 큰 계획 문서
  - 관련 모듈 `README.md`
  - 필요하면 현재 truth가 있는 `docs/logbook.md`
- 큰 계획 문서도 현재 살아 있는 작은 실행 체크리스트 문서를 링크로 모아, 문서를 타고 들어가면 자연스럽게 `큰 계획 -> 작은 실행 -> 관련 기준 문서` 흐름이 보이게 유지한다.
- 날짜 prefix 문서에서는 가능하면 파일명만 텍스트로 적지 말고 markdown 링크로 걸어, 나중에 다시 들어왔을 때 문서를 타고 타고 이동할 수 있게 만든다.
- 실제 구현 중에는 큰 단계가 아니라 `작은 실행`을 시작할 때마다 해당 체크리스트 문서를 다시 읽고, 완료한 항목은 즉시 체크하거나 상태를 갱신한다.
- 작업이 끝날 때는 코드만 끝내지 않고, 체크리스트 문서와 관련 canonical 문서를 함께 갱신해 나중에 다시 들어와도 `무엇이 끝났고 무엇이 다음으로 연결되는지` 복구 가능하게 남긴다.
- 장시간 변환, calibration 입력 준비, 실기기 benchmark 파이프라인처럼 여러 날 이어질 수 있는 작업은 `계획 변경 이력`도 같이 남긴다.
- 즉 최초 계획 문서만 만들고 끝내지 않고, 중간에 양자화 옵션, 모델 구조, calibration 정책, smoke 범위가 바뀌면 같은 계획 문서나 연결된 logbook에 바로 기록한다.

### 7-4A. 실행 가능한 코드는 자기 안에 재현 명령을 남긴다

- 앞으로 새로 만드는 변환, 검증, benchmark, 배포 준비 코드는, 사람이 Codex 없이 다시 돌릴 수 있게 `실행 메모`를 코드 안에 직접 남긴다.
- 별도 메모 파일로만 남기지 말고, 가능한 한 해당 실행 파일의 상단 docstring 또는 주석에 아래를 같이 적는다.
  - smoke 실행 명령
  - full 실행 명령
  - 자주 바꾸는 주요 인자 의미
  - 입력 경로와 출력 경로
- helper 함수 파일처럼 직접 실행하지 않는 코드라면, 그 파일이 아니라 실제 entry script 쪽에 실행 메모를 남긴다.
- 코드가 정상 실행되는 것이 확인되면, 그때 실제로 사용한 터미널 입력과 핵심 인자를 코드 주석에 최신 기준으로 반영한다.
- 목적은 아래 셋이다.
  - Codex 없이도 재현 가능한 실행 기반 유지
  - 장시간 변환과 실기기 점검 자동화 시 사람 handoff 비용 감소
  - 나중에 문서만 보고 코드 실행 경로를 다시 추적하는 시간 절감

### 7-4B. smoke와 실패 산출물은 자동 삭제한다

- smoke 테스트는 반드시 수행하지만, smoke 산출물은 최종 canonical 산출물이 확보되면 자동 삭제한다.
- 실패 run, 중간 retry run, 원인 분석용 임시 산출물도 문제 원인과 수정 사항이 문서에 남는 즉시 자동 삭제한다.
- 즉 리포 안에는 기본적으로 `현재 기준으로 다시 써야 하는 canonical 산출물`만 남긴다.
- smoke와 실패 산출물은 코드, 실행 명령, 보고서 요약 수치로만 재현 가능해야 하며, 결과물 폴더 자체를 역사 자산처럼 계속 보관하지 않는다.
- 삭제 전에는 아래를 먼저 확인한다.
  - 최종 canonical 산출물이 이미 존재하는지
  - 관련 보고서와 logbook에 핵심 수치와 원인 분석이 남았는지
  - active 문서가 삭제 대상 경로를 current truth처럼 가리키고 있지 않은지
- 삭제를 수행하면 같은 턴에 아래를 같이 반영한다.
  - `docs/logbook.md` 또는 해당 모듈 `docs/logbook.md`에 삭제 일시와 삭제 대상 목록 기록
  - 관련 `README.md`, active 보고서, current truth 문서에서 삭제된 경로 제거 또는 삭제 사실 명시
- 결과적으로 active 문서를 따라가면 `살아 있는 산출물만 링크되고`, 삭제된 smoke/failed 자산은 로그에서만 이력을 볼 수 있어야 한다.

### 7-5. 진행 보고는 짧고 자주 한다

- 장시간 작업에서는 주요 액션 2~3개마다 짧은 진행 보고를 남긴다.
- 보고 내용은 기본적으로 `무엇을 확인했는지`, `어디서 막혔는지`, `다음에 무엇을 할지`를 포함한다.
- 사용자는 터미널 원문을 보지 못하므로, 긴 실행에서는 핵심 출력과 의미를 짧게 번역해 같이 전달한다.

### 7-6. 장시간 작업은 마일스톤마다 commit 여부를 점검한다

- 이 프로젝트에서는 사소한 수정까지 포함해 변경이 생기면 가능한 한 같은 턴 안에 commit과 push를 끝낸다.
- 큰 작업뿐 아니라 작은 문서 수정, 경로 정리, 스크립트 보강처럼 자주 잊히는 변경도 local에 오래 남겨두지 않는다.
- 기본 흐름은 `git add -A -> git commit -> configured remotes push`다.
- 반복 실행을 위해 공용 동기화 스크립트를 두고, 사람이 직접 순서를 잊지 않도록 한다.
- commit 메시지는 한국어 중심으로 쓰되, 모듈명, 모델명, 라이브러리명, 경로처럼 식별성이 중요한 기술 키워드는 영어 그대로 유지해도 된다.
- 커밋할 변경이 없더라도 push 누락 여부는 점검한다.

### 7-7. 터미널 정리는 보수적으로 한다

- 터미널이나 백그라운드 프로세스를 정리할 때는 `지금 당장 눈에 띄는 용도 없음`만으로 종료하지 않는다.
- 종료 전에는 최소한 `ps`, `tty`, 진행률 파일, 상태 파일, 출력 경로`를 확인해 실제 역할을 먼저 판별한다.
- 하나라도 불확실하면 종료하지 않고 유지 쪽으로 보수적으로 판단한다.

## 8. 장시간 작업과 watchdog 규칙

- `10분 이상` 걸릴 가능성이 있거나 `여러 단계가 자동으로 이어지는` 작업은 watchdog 대상으로 본다.
- 실제 대규모 실행 전에 watchdog smoke를 먼저 통과시킨다.
- `시간을 아끼기 위해 smoke를 생략`하지 않는다. 이 프로젝트에서는 smoke가 시간을 버는 행동으로 본다.
- smoke는 최소 아래 셋을 검증한다.
  - 정상 종료
  - 비정상 종료
  - heartbeat 또는 progress가 멈춘 stale 상태
- smoke는 가능하면 `가장 작은 실제 경로`를 검증한다.
  - 예: 1개 모델, 2~3장 얼굴 이미지, 짧은 calibration 묶음, 1회 실기기 benchmark
- smoke를 통과하면 바로 full run으로 복사할 수 있게, smoke 명령과 full 명령의 차이는 `입력 수`, `calibration 수량`, `benchmark 반복 횟수`, `경로` 정도로만 유지한다.
- smoke를 통과하기 전에는 실제 대규모 변환이나 장시간 benchmark를 걸지 않는다.
- watchdog는 최소 아래 산출물을 남겨야 한다.
  - `job.spec.json`
  - `job.pid`
  - `status.local.md`
  - `events.log`
  - `stdout.log`
  - `stderr.log`
  - 종료 코드 또는 종료 원인
  - 다음 단계 실행 여부
- 변환이나 benchmark가 끝났다고 해서 바로 다음 단계에 들어가지 않는다.
- watchdog가 `실제 본체 종료`와 `산출물 완결성`을 확인한 뒤 다음 단계로 넘긴다.
- monitor만 남고 본체가 죽은 경우는 실패로 기록한다.

## 9. 산출물과 자산 운영 원칙

### 9-1. 로컬 워크스페이스 구조

- 로컬 워크스페이스는 기본적으로 `repo / envs / secrets` sibling 구조를 유지한다.
- 민감한 운영 정보는 리포 바깥 `../secrets/`에 둔다.
- 비공개 원격, 비공개 자산 업로드/동기화, 자격증명, 외부 미러 운영 세부는 tracked repo가 아니라 `../secrets/README.local.md`에서만 관리한다.

### 9-2. 실행 산출물은 모듈 내부 `results/`만 사용한다

- 실제 대용량 실행 산출물은 각 모듈 내부 `<module>/results/` 아래에만 둔다.
- root `results/`는 만들지 않는다.
- 모듈 내부 첫 단계 폴더명은 항상 `YYMMDD_HHMM_설명` prefix를 사용한다.
- 단, 하나의 장기 작업 안에서 canonical 데이터셋/모델 폴더 아래에 단계별 산출물이 여러 개 생기면, 그 하위 단계 폴더는 `01_`, `02_`, `03_`처럼 실행 순서 숫자 prefix를 붙인다.
- 즉 `results`의 실행 루트는 날짜 prefix, 그 루트 안이나 canonical 데이터셋 트리 안의 단계 폴더는 숫자 prefix를 기본 규칙으로 함께 쓴다.
- smoke, debug, failed run 산출물은 결과가 문서로 승격된 뒤 삭제 가능한 임시 자산으로 본다.

### 9-3. 추적 자산과 로컬 자산을 구분한다

- git으로 추적할 가치가 있는 소형 자산과 metadata만 repo 안에 둔다.
- 대용량 실행 파일, 체크포인트, 대규모 데이터셋, 장치 종속 프로파일 산출물은 로컬 경로 또는 `.gitignore` 경로에 둔다.
- 단, 재현에 꼭 필요한 소형 metadata, manifest, dataset card는 repo 안에 둘 수 있다.

### 9-4. 새 산출물 생성 전에는 디렉토리 인벤토리와 기준 문서를 먼저 확인한다

- 코드 실행, 문서 작성, 파일 복사/이동, 다운로드, 변환, 아카이빙처럼 실제로 새 파일이나 폴더를 만들 가능성이 있는 작업 전에는 먼저 아래 명령을 실행한다.
  - `python tools/directory_inventory.py --module <module> --kind <kind> --candidate-name <name>`
- 목적은 아래 셋이다.
  - 기존 canonical 디렉토리를 재사용할 수 있는지 먼저 확인
  - 중복되거나 의미가 겹치는 새 폴더 생성을 방지
  - 나중에 삭제 판단을 어렵게 만드는 혼동성 이름을 미리 차단
- 스크립트가 출력한 기준 문서는 대충 훑지 말고 끝까지 읽는다.
- 새 문서나 산출물을 만들기 전에는 기존 기록과 비교해 중복과 누락 여부를 먼저 판단한다.
- 스크립트 결과를 보고도 기존 경로가 맞지 않을 때만 새 폴더를 만든다.
- 프로젝트 초기 부트스트랩처럼 아직 `tools/directory_inventory.py`가 없는 단계라면, 먼저 `find`와 `rg --files`로 shallow inventory를 확인하고 그 사실을 `docs/logbook.md`에 기록한다.
- 같은 phase 안에서 새 폴더 생성 규칙이 반복되기 시작하면, 가능한 한 그 phase 안에 인벤토리 스크립트를 만들고 이후 기준 절차로 승격한다.
- 현재 스크립트로 판단이 부족하면 새 파일을 만들기 전에 인벤토리 스크립트를 먼저 보강하거나 새로 만든다.
- 인벤토리 스크립트는 raw/extracted/cache 같은 대규모 트리를 재귀 탐색하지 않고, shallow listing과 manifest/README 기반 판단을 기본으로 유지한다.

## 10. 문서 운영 원칙

### 10-1. 공용 문서는 최소화한다

- `docs/`에는 공용 운영 문서와 공용 logbook만 남긴다.
- 프로젝트 레벨에서 오래 유지할 공용 단일 문서는 아래 셋만 둔다.
  - `docs/AGENT.md`
  - `docs/logbook.md`
  - `README.md`

### 10-2. 모듈별 상세 문서는 모듈로 내린다

- 모듈별 상세 아키텍처, 고정 결정, 경로, 실행 방법은 각 모듈 `README.md`에 상세하게 기록한다.
- 모듈별 현재 상태와 최근 기록은 각 모듈 `docs/logbook.md`를 기본 기준으로 둔다.
- 시간이 지나며 누적되는 환경/보고서 문서는 각 모듈 `docs/` 아래에 둔다.
- 자산 스크린샷, GIF, 동영상, 컨셉 이미지 같은 문서 자산도 가능하면 그 자산을 실제로 사용하는 모듈 `docs/assets/` 아래에 둔다.
- 특정 모듈에 속하지 않는 프로젝트 전역 그림, 공용 구조 이미지처럼 root `README.md`가 직접 쓰는 자산은 `assets/` 아래에 둔다.

### 10-2-3. 모듈 문서는 `README + logbook`를 기본으로 유지한다

- 모듈 안에서 자주 바뀌는 작은 작업 기록은 새 `보고서` 파일을 계속 만드는 대신 기본적으로 `docs/logbook.md`에 누적한다.
- 비사소한 작업의 체크리스트도 기본 위치는 `docs/logbook.md`다.
- 즉 `docs/logbook.md`는 `현재 상태 + 최근 로그`만 적는 문서가 아니라, 현재 작은 실행의 활성 체크리스트를 함께 두는 기본 운영 문서로 본다.
- 모듈 `README.md`는 안정된 설명과 현재 고정 결정만 남기고, 실행 중인 세부 단계 기록까지 모두 끌어안지 않는다.
- 에이전트는 스스로 새 `보고서`를 만들지 않는다.
- `docs/보고서/`는 사용자가 별도로 요청했거나 외부 공유가 필요한 경우에만 아래처럼 제한해서 쓴다.
  - 마일스톤 요약
  - 후보 비교 결과
  - 외부 공유용 정리 문서
- 보고서는 새 로그를 쌓는 자리가 아니다. 보고서의 재료는 항상 `README.md`와 `docs/logbook.md`에 이미 남아 있는 current truth와 로그여야 한다.
- 별도 `보고서`로 승격한 경우에도, 현재 무엇을 기준으로 읽어야 하는지는 `docs/logbook.md`가 먼저 가리키게 유지한다.
- 작은 단계가 끝나면 핵심 결과는 `README.md`와 `docs/logbook.md`에 먼저 반영하고, 완료된 상세 체크리스트는 더 이상 active current truth로 취급하지 않는다.
- 오래된 상세 문서는 바로 삭제하지 말고, 먼저 `README + logbook`에 핵심을 옮긴 뒤 archive나 historical 문서로 내린다.

### 10-2-4. 로그는 logbook에만, README는 현재 상태만 둔다

- 로그성 데이터, 실행 이력, 시행착오, 임시 판단, 중간 수치, 삭제 기록은 모두 `docs/logbook.md`에만 남긴다.
- `README.md`에는 현재 상태, 현재 고정 결정, 현재 구조, 현재 실행 절차만 남긴다.
- 즉 `README.md`를 changelog처럼 쓰지 않는다.
- 보고서를 작성할 때도 새로운 로그를 만들지 않고, 기존 logbook에 쌓인 내용을 사용 요청에 맞게 다시 정리한다.
- 보고서 안에 로그성 데이터가 들어가 있으면, 먼저 logbook에 옮기고 보고서는 요약만 남기도록 리팩토링한다.

### 10-2-1. root `README.md` 아래에는 프로젝트 전역 고정 메모를 누적할 수 있다

- 모듈별로 나눌 수 없는 프로젝트 전역 구조 메모, 전체 아키텍처 설명, 공용 자산 설명은 `README.md` 아래쪽에 누적한다.
- 단, 최근 고민, 임시 메모, 당일 진행 상황처럼 시간이 지나며 빠르게 바뀌는 내용은 `README.md`가 아니라 `docs/logbook.md`에 둔다.
- 즉 `README.md`는 처음 방문한 사람이 읽는 소개 문서이면서, 동시에 프로젝트 전역의 안정된 흔적을 남기는 자리다.

### 10-2-2. 문서 반영 방침 자체도 기억에 맡기지 않는다

- 문서 구조, 경로 규칙, 자산 ownership, archive 규칙, naming 규칙, 읽는 순서, 생성 전 preflight 규칙처럼 앞으로 계속 지켜야 하는 기준이 새로 생기면, 그 규칙을 기억에 맡기지 않고 같은 턴 안에 canonical 문서에 바로 반영한다.
- 기본 반영 대상은 아래와 같다.
  - 운영 규칙이면 `docs/AGENT.md`
  - 현재 truth와 전역 결정이면 `docs/logbook.md`
  - 프로젝트 전역 설명이면 `README.md`
  - 특정 모듈 상세 기준이면 해당 모듈 `README.md`
- 경로를 옮기거나 새 기준 디렉토리를 만들었는데도 문서를 갱신하지 않은 상태로 작업을 끝내지 않는다.
- 규칙을 반복해서 지켜야 하는데 사람이 잊기 쉬운 성격이면, 문서만 적고 끝내지 않고 가능한 한 스크립트나 체크 절차도 같이 만든다.
- 사용자가 운영 방침을 추가하거나 바꾸면, 그 변경도 같은 턴 안에 canonical 문서와 필요한 스크립트에 바로 반영한다.
- 즉 이 프로젝트에서는 기억보다 문서와 자동화 스크립트를 더 신뢰한다.

### 10-3. logbook는 단일 기록 문서로 운영한다

- 프로젝트 레벨의 현재 상태, 전역 결정, 최근 로그는 `docs/logbook.md` 하나로 유지한다.
- 예전의 `status`, `decisions` 역할은 `logbook` 상단 고정 섹션으로 흡수한다.
- active logbook를 읽을 때는 항상 최신 archive 1개를 함께 읽어 최근 맥락을 복구한다.

### 10-4. logbook archive 규칙

- 새 로그를 쓰기 전에는 기본적으로 아래 명령으로 모든 active logbook를 한 번에 점검한다.
  - `python tools/logbook_archive_all.py --archive-if-needed`
- 특정 logbook 하나만 점검해야 할 때는 아래 명령을 쓴다.
  - `python tools/logbook_archive_guard.py --logbook <path>/docs/logbook.md --archive-dir <path>/docs/logbook_archive --archive-if-needed`
- 부트스트랩 단계에서 archive 스크립트가 아직 없다면, 해당 턴에는 `wc -l`로 줄 수를 직접 확인하고 필요해지는 시점에 스크립트를 추가한다.
- active logbook 줄 수가 `1000`을 넘으면 현재 파일을 아래 형식으로 archive한다.
  - `docs/logbook_archive/logbook_YYMMDD_HHMM.md`
- 모듈 logbook도 같은 규칙을 따른다.
  - `<module>/docs/logbook_archive/logbook_YYMMDD_HHMM.md`
- archive 뒤 active logbook는 아래 고정 섹션만 남긴 채 다시 시작한다.
  - `읽기 규칙`
  - `현재 프로젝트 스냅샷`
  - `현재 전역 결정`
  - `현재 활성 체크리스트`
  - `Recent Logs`

### 10-5. 날짜 prefix 규칙

- 아래처럼 시간이 지나며 계속 쌓이는 문서는 반드시 날짜 prefix를 붙인다.
  - `<module>/docs/보고서/*.md`
  - `<module>/docs/환경/*.md`
  - `docs/logbook_archive/logbook_*.md`
- 형식은 아래로 고정한다.
  - `YYMMDD_HHMM_이름.md`
- 예: `260324_1103_이름.md`
- 고정 파일명 예외는 아래처럼 단 하나의 최신 기준만 유지하는 문서다.
  - `README.md`
  - `docs/AGENT.md`
  - `docs/logbook.md`
  - `<module>/README.md`
  - `<module>/docs/README.md`
  - `<module>/results/README.local.md`

### 10-6. 새 문서를 기록할 때는 관련 기존 문서를 정독한다

- 문서를 새로 쓰거나 큰 폭으로 갱신할 때는 관련 기존 문서를 대충 훑지 않는다.
- 양이 많아도 끝까지 읽고, 아래 셋을 꼼꼼히 확인한다.
  - 이미 같은 결론이 다른 문서에 적혀 있는지
  - 이번 문서에 빠진 사실이나 경로가 없는지
  - 역할이 다른 문서에 들어가야 할 내용을 잘못 끌어오지 않았는지
- 특히 `README`, `logbook`, `환경`, `보고서` 문서는 작성 전에 기존 canonical 문서와 최신 archive를 함께 읽고 판단한다.
- 문서 정리는 속도보다 무중복과 무누락을 우선한다.

### 10-7. 문서가 없는 프로젝트 부트스트랩

- 새 프로젝트나 비어 있는 디렉토리에서 실제 개발을 시작하는데 문서가 전혀 없다면, 최소 아래 문서부터 만든다.
  - `docs/AGENT.md`
  - `docs/logbook.md`
  - `README.md`
- 부트스트랩 직후에는 현재 주경로를 대표하는 최소 모듈 `README.md`와 모듈 `docs/logbook.md`도 함께 만든다.
- 이 프로젝트의 초기 기본 모듈은 우선 `conversion/`과 `runtime/`으로 본다.
- 단, 단발성 한 파일 수정이나 짧은 임시 실험처럼 프로젝트 운영 체계가 필요 없는 작업에는 이 규칙을 강제하지 않는다.

## 11. 작업 시작 체크리스트

1. `docs/AGENT.md`를 다시 읽는다.
2. `README.md`를 읽고 프로젝트 개요를 맞춘다.
3. `docs/logbook.md`를 읽고 현재 스냅샷과 전역 결정을 확인한다.
4. 최신 `docs/logbook_archive/logbook_*.md` 1개를 읽고 가장 최근 맥락을 복구한다.
5. 관련 모듈 `README.md`를 읽고 해당 모듈의 상세 고정 결정을 확인한다.
6. 새 파일이나 폴더를 만들 일이라면 `python tools/directory_inventory.py --module <module> --kind <kind> --candidate-name <name>`를 먼저 실행한다. 아직 스크립트가 없다면 `find`와 `rg --files`로 shallow inventory를 확인하고 그 사실을 `docs/logbook.md`에 남긴다.
7. 비사소한 개발 작업이라면 시작 전에 체크리스트 기반 실행 문서가 있는지 확인하고, 없거나 현재 작은 실행 단위를 충분히 설명하지 못하면 먼저 만든다.
8. 이때 기본 체크리스트 위치는 관련 모듈 `docs/logbook.md`로 보고, 큰 단계나 장시간 작업만 별도 `docs/보고서/` 문서로 올린다.
9. 작은 실행을 시작하기 직전에는 해당 체크리스트를 다시 읽고, 이번에 수정할 파일, 검증 방법, 산출물 연결점을 다시 맞춘다.
10. 지금 하려는 작업이 현재 단계 기준으로 오버엔지니어링 금지 원칙에 어긋나지 않는지 먼저 점검하고, 과하면 더 작은 주경로 작업으로 줄인다.
11. `docs/logbook.md`에 새 로그를 쓸 일이라면 `python tools/logbook_archive_guard.py --archive-if-needed`를 먼저 실행한다. 아직 스크립트가 없다면 `wc -l`로 줄 수를 확인하고 수동 판단 사실을 남긴다.
12. 이번 턴에서 새로 생긴 규칙, 경로, ownership, 운영 기준이 있다면 종료 전에 canonical 문서와 필요한 스크립트까지 같이 갱신한다.
13. 변경된 파일이 있으면 종료 전에 `tools/git_sync_all.sh "<커밋 메시지>"` 또는 같은 절차로 commit과 push를 끝낸다. 아직 git 저장소가 초기화되지 않았거나 remote가 없다면, 그 상태를 `docs/logbook.md`에 남기고 초기화 TODO를 활성 체크리스트에 올린다.
